Candidate Switching with CYOA

Hi hi people, welcome back to the blog! Sorry for the break last week - I had a lot of stuff going on in both my job and my personal life, and I didn’t want to throw out anything half-baked. I’ll try to keep these weeks where I publish nothing to a minimum, but I can’t make any promises. Or I’ll just write some posts that are more personal opinion stuff and less technical guides, since these are a lot faster to write when I don’t have a lot of time. We’ll see.

Anyway, my plan for this week was to give some guidance for a bunch of little CYOA helper functions I posted on the TCT Discord a while back. However, I realized that while these are neat, it’s not the most common thing people want to use when they add CYOA to their mod. Ever since mods like 1968Romney and 1964Midnight did it, candidate and VP switchers are the big thing that can be done with CYOA (though the North Korea mod and I think an updated version of 1972 did it first, technically). So let’s dive in, how can you do that?

One of the earliest examples of a candidate switcher - and what a great way to utilize it!

I’ll go over three different cases here that are relevant: Switching the player VP, the opponent candidate and the player candidate. I will not cover how to switch the opponent VP - for the simple reason that, unless you want to use a fancy results table like in Peace with Honor for example, the opponent’s VP doesn’t show up anywhere. In fact, I don’t even know if the game per default knows which VP your opponent uses (probably, it defaults to the first one defined or something).

The player VP, however, makes a difference. Still, even here, it mostly makes a difference before the game proper starts - depending on the VP selected, a different Code 2 is loaded, with (potentially) different questions, effects, CYOA and so on. Once the game has started, the VP mostly influences two things - the name and picture shown at the bottom of the screen. Additionally, the VP has a home state in the code which I assume gives a small bonus in that state, but I never actually checked. The running mate can also change the issue score values of your candidate.

So, to change the VP all we need to do is change the name, picture and homestate then. With what we learned two weeks ago about the structure of the game code, that’s easy-peasy, right? Should go something like this:

campaignTrail_temp.running_mate_image_url = "new_link.jpg"
campaignTrail_temp.running_mate_last_name = 'Whitmer'
campaignTrail_temp.running_mate_state_id = pk_of_a_state

That’s it! Put these three lines in an appropriate spot in your CYOA function (see earlier blogpost on how to determine when the game should run the switch), and the VP will change after the next question. Changing the running_mate_issue_scores works the same way, it’s just a little more complex since they are an array of objects in itself, not just a single value like the stuff above. The easiest way to change them is to probably swap out the whole array.
Also of note, this doesn’t change any questions - you’ll need to use some different code to achieve that. (Which, you know what I’m going to say, I might cover another time)

Exchanging opponents works a little differently. Here, you need to change the name in a completely different part of the code - you don’t see your opponents name at the bottom of your screen, but on the map, in the ending area… Luckily, for all those spots, the name is drawn from the same source, so you just need to change it once:

campaignTrail_temp.candidate_json.find(candidate => candidate.pk ==pk_of_opponent).fields.first_name = "Nikki"
campaignTrail_temp.candidate_json.find(candidate => candidate.pk ==pk_of_opponent).fields.last_name = "Haley"
campaignTrail_temp.candidate_json.find(candidate => candidate.pk == pk_of_opponent).fields.color_hex = ‘hexcolourcode’

Looking complicated? Let’s get through it step by step: You take the object campaignTrail_temp (see last post), then traverse to the object candidate_json.That contains an array, in which you find the candidate with the pk you want to change. That candidate is an object again - you move to the sub-object fields and then to the variable first_name/last name/color_hex.

All this is possible with candidate switching and some variable tracking and conditional logic.

Don’t worry if you don’t understand all that in detail - it’s great if you do, since you can then use similar code to change similar stuff. But if you don’t understand the code in detail, you can still just use it, and it will work fine. In any case, you can do 1920/PwH/W.-esque shenanigans with that code and the one from the blog post a while ago that talks about conditions.

But something is still missing - we’ve gone over changing VPs and opponents, but how do you change your candidate (Note - by changing candidate, I mean like in 68Romney. I don’t mean switching from Hillary to Trump in 2016, for example. If that’s possible, it’s significantly more complicated). Luckily, you already know how to switch your candidate, even if all your knowledge about CYOA comes from this code. How so? Switching your candidate works like this:

campaignTrail_temp.candidate_image_url = "new_link.jpg"
campaignTrail_temp.candidate_last_name = 'Whitmer'
campaignTrail_temp.candidate_json.find(candidate => candidate.pk ==pk_of_player).fields.first_name = "Gretchen"
campaignTrail_temp.candidate_json.find(candidate => candidate.pk ==pk_of_player).fields.last_name = "Whitmer"
campaignTrail_temp.candidate_json.find(candidate => candidate.pk == pk_of_player).fields.color_hex = ‘hexcolourcode’

Looking familiar? You’re basically just chaining the VP switcher and opponent switcher, with some small adjustments. But it basically works the same way!
Of course, all this code can also be combined (so the candidate and the VP change at the same time), or modularized (so, for example, only the image is changed, or only the colour a candidate uses). Lots of possibilities, and as you see, the code is not that complicated.

Just changing the names and pictures (along with the theme) can lead to cool effects like this.

That wraps it up for today - have fun with the candidate switching! Next time, expect some thoughts about game design in the context of TCT and after that, more code to do cool CYOA stuff. Until then, have a good time!
